# ADR-008 â€” The operational facade is generated from canonical sources

- Status: accepted
- Date: 2026

## Context

The repository already demonstrates the Managed-File rule on the *data* plane:
the SQLite index is derived from the canonical log, regenerated by `rebuild`,
and reconciled by `verify`. Hand-editing the index is forbidden because it is a
projection, not a source.

The *operational* plane had the same intent but no mechanism. Agent personas,
prompts, and skills lived directly under `.github/`, hand-authored in the place
an agent runtime loads them. The governance boundary instruction said *"never
patch a generated mirror; regenerate it"* â€” but nothing generated them, and
nothing checked them. That is precisely the failure the governance map warns
about: a "MUST" that no executable check enforces is aspirational. Half of the
Managed-File rule was real (the index); half was a promise (the mirrors).

A faithful reference architecture cannot have an enforced data plane and an
unenforced operational plane. Either the mirror rule is mechanised, or the docs
are overclaiming.

## Decision

1. **Canonical sources move to the authority plane.** Agent personas, prompts,
   and skills are authored once under `data/agents/`:
   - `data/agents/*.agent.md`, `data/agents/README.md` â†’ `.github/agents/`
   - `data/agents/prompts/**` â†’ `.github/prompts/`
   - `data/agents/skills/**` â†’ `.github/skills/`
   The `.github/` copies are **generated mirrors**, not sources.

2. **A generator regenerates the mirrors.** `src/governed_memory/mirror.py`
   maps every canonical source to its mirror path and rewrites stale or missing
   mirrors, pruning orphans whose canonical source was deleted. It is the
   operational-plane analogue of `rebuild`.

3. **A drift check gates the mirrors.** `governed-memory sync-mirrors --check`
   reports drift and exits non-zero without writing â€” the operational-plane
   analogue of `verify`. A mirror is correct only if it is byte-identical to its
   canonical source.

4. **Edit canonical, never the mirror.** A human edits `data/agents/...` and
   regenerates. Patching a file under `.github/agents|prompts|skills` directly is
   the same category of error as hand-editing the SQLite index.

## Consequences

- The Managed-File rule is now demonstrated on both planes with the same shape:
  *canonical source â†’ generator â†’ derived artifact â†’ drift check*. The data
  plane uses `rebuild`/`verify`; the operational plane uses `sync-mirrors` /
  `sync-mirrors --check`.
- The mirror language in the governance boundary instruction is no longer
  aspirational â€” there is an entrypoint that regenerates and a check that gates.
- A reader copying this structure inherits a worked example of generating an
  operational facade, not just a folder of hand-maintained files.
- The mirror mapping is deliberately small and deterministic. A real deployment
  (the kind this mirrors) has many more agents across many domains and runs the
  same regeneration in a pre-commit hook; the mechanism is identical, only the
  inventory grows.

## Enforcement

- `mirror.py` is the regeneration entrypoint; `sync-mirrors --check` is the
  drift-check. Both are declared for the mirror Managed Files in
  [`ownership-taxonomy.md`](../../../.github/ownership-taxonomy.md).
- `tests/test_mirror.py` proves the round trip: canonical â†’ regenerated mirror
  is byte-identical, an edited mirror is reported as drift, and a deleted
  canonical source prunes its orphan.
